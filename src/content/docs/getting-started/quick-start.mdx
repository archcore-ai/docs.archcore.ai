## Install

<Tabs>
  <TabItem label="macOS / Linux">
    ```bash
    curl -fsSL https://archcore.ai/install.sh | bash
    ```
  </TabItem>
  <TabItem label="Windows">
    Download `archcore.exe` from the <a href="https://github.com/archcore-ai/cli/releases/latest" target="_blank">latest release</a> and add it to your `PATH`.

    ```powershell
    # Example: move to a directory in your PATH
    Move-Item archcore.exe C:\Users\$env:USERNAME\.local\bin\
    ```
  </TabItem>
  <TabItem label="Windows (WSL)">
    Install <a href="https://learn.microsoft.com/en-us/windows/wsl/install" target="_blank">WSL</a>, then run inside it:

    ```bash
    curl -fsSL https://archcore.ai/install.sh | bash
    ```
  </TabItem>

## Initialize Your Project

```bash
cd your-project
archcore init
```

This does three things:
1. Creates `.archcore/` directory with `settings.json`
2. Auto-detects your coding agents (Claude Code, Cursor, Copilot, etc.)
3. Installs MCP server config and session hooks for detected agents

**Expected result:** After this, your agent can list documents, read them, and create structured architectural docs. If no agents are detected, Archcore falls back to configuring Claude Code.

## Create Your First Document

The easiest way is to ask your AI agent:

> "Create an ADR about using PostgreSQL for our database"

The agent will use the `create_document` MCP tool and generate a document from the ADR template with the proper structure, frontmatter, and required sections.

You can also create documents manually:

```bash
mkdir -p .archcore/database
```

Create `.archcore/database/use-postgres.adr.md`:

```markdown
---
title: Use PostgreSQL as Primary Database
status: accepted
---

## Context

We need a reliable relational database for our application.

## Decision

Use PostgreSQL for all persistent data storage.

## Alternatives Considered

- MySQL — fewer advanced features (JSONB, arrays, CTEs)
- MongoDB — doesn't fit our relational data model

## Consequences

### Positive
- Strong ACID guarantees
- Excellent ecosystem and tooling
- JSONB for semi-structured data when needed

### Negative
- Requires schema migrations for changes
- Horizontal scaling is more complex than NoSQL
```

## Validate

Check that everything is structured correctly:

```bash
archcore validate
```

## Use With Your Agent

Start a conversation with your coding agent. It now has access to your `.archcore/` documents through MCP.

Try asking:
- *"What architectural decisions have been made?"*
- *"Create a rule for database migrations based on the PostgreSQL ADR"*
- *"List all documents and their relations"*

The agent should respond with information from your `.archcore/` documents. If it doesn't see them, check the [troubleshooting guide](/troubleshooting/agent-not-seeing-documents/).

## Starting From an Existing Repo

Already have project documentation? Archcore is additive — it works alongside your existing files.

**You have a `CLAUDE.md` or `.cursorrules`:**
Keep it. Over time, extract structured knowledge (decisions, rules, standards) into typed `.archcore/` documents. Your flat file can reference them or shrink as Archcore takes over.

**You have `docs/` or existing ADRs:**
Ask your agent to help migrate: *"Migrate our existing documentation into Archcore format."* The agent will read your files, pick the right document types, and create properly structured `.archcore/` documents through MCP.

**You have a wiki or Notion:**
Start by capturing the most frequently referenced decisions and rules as `.archcore/` documents. You don't need to migrate everything at once — start with what agents need most.
