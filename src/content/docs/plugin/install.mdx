The plugin runs inside your coding agent (Claude Code or Cursor). For Claude Code, no prerequisites — the plugin bundles a launcher that resolves the Archcore CLI on first use (from an existing global install, a plugin-managed cache, or a checksum-verified download). MCP is registered automatically.

For Cursor, you register MCP yourself — see the Cursor tab below.

## Install the plugin

<Tabs>
	<TabItem label="Claude Code">
		From your shell:

		```bash
		claude plugin marketplace add archcore-ai/archcore-plugin
		claude plugin install archcore@archcore-plugins
		```

		Or, from within Claude Code:

		```bash
		/plugin marketplace add archcore-ai/archcore-plugin
		/plugin install archcore@archcore-plugins
		```

		That's it. Open your project and run `/archcore:help` — the first Archcore tool call triggers a one-time CLI download (~5s, cached between sessions), and initializes `.archcore/` through MCP if missing.

		**Already have `archcore` on `PATH`?** The bundled launcher defers to any global install — your existing binary is used, no duplicate cache, no conflict.
	</TabItem>
	<TabItem label="Cursor">
		Requires Cursor 2.5+. Archcore isn't on the official Cursor Marketplace yet, so install it straight from GitHub — open a new Agent chat and run:

		```
		/add-plugin archcore@https://github.com/archcore-ai/archcore-plugin
		```

		Type the full command — `/add-plugin` doesn't appear in autocomplete. For team rollouts, paste the same GitHub URL under Dashboard → Settings → Plugins → Team Marketplaces → Import.

		Then register MCP in Cursor's MCP settings (or a project `mcp.json`). Point the `command` at the plugin's bundled launcher:

		```json
		{
			"mcpServers": {
				"archcore": {
					"command": "/abs/path/to/archcore-plugin/bin/archcore",
					"args": ["mcp"]
				}
			}
		}
		```

		Alternatively, install the CLI globally and point `command` at `archcore`. Either way, the resolved binary is the same.

		Cursor support is implemented but has a slightly different hooks API — see [Supported Hosts](/plugin/supported-hosts/).
	</TabItem>
	<TabItem label="Local development (any host)">
		Clone the plugin repo:

		```bash
		git clone https://github.com/archcore-ai/archcore-plugin
		```

		**Claude Code** — load the cloned directory for the session:

		```bash
		claude --plugin-dir /path/to/archcore-plugin
		```

		**Cursor** — Cursor has no `--plugin-dir` flag. Symlink the repo into Cursor's local plugins directory and reload the window:

		```bash
		ln -s /path/to/archcore-plugin ~/.cursor/plugins/local/archcore
		# then in Cursor: Cmd/Ctrl+Shift+P → "Developer: Reload Window"
		```

		Both manifests (`.claude-plugin/plugin.json` and `.cursor-plugin/plugin.json`) sit at the repo root.

		Local mode is useful when hacking on skills or hooks. The launcher still auto-resolves the CLI — set `ARCHCORE_BIN=/abs/path/to/your/archcore` to test against a locally-built CLI binary.
	</TabItem>
</Tabs>

## Offline / enterprise / BYO CLI

If you already have the Archcore CLI installed globally, the launcher respects it — the `archcore` on `PATH` always wins over the plugin-managed cache.

For fully offline environments, set `ARCHCORE_SKIP_DOWNLOAD=1` to disable the launcher's auto-download. The launcher then falls through to `ARCHCORE_BIN` or `PATH` only. Alternatively, pin an explicit binary with `ARCHCORE_BIN=/abs/path/to/archcore`.

### Plugin environment variables

| Variable                     | Effect                                                                       |
| ---------------------------- | ---------------------------------------------------------------------------- |
| `ARCHCORE_BIN`               | Pin an explicit `archcore` binary; wins over PATH and the cache.             |
| `ARCHCORE_SKIP_DOWNLOAD`     | `=1` disables the launcher's auto-download (offline mode).                   |
| `ARCHCORE_DISABLE_INJECTION` | `=1` turns off the pre-edit context injection hook globally.                 |

See the [CLI install guide](/cli/install/) for ways to install the CLI manually (curl, PowerShell, `go install`, from source).

## Verify

1. Restart your coding agent so the session-start hook can run.
2. In a fresh conversation, run:

	```
	/archcore:help
	```

	You should see a list of available skills and intent commands.
3. Try an intent command on a real decision:

	```
	/archcore:decide use PostgreSQL as the primary database
	```

	The agent picks the right document type (an ADR), creates a draft in `.archcore/` via MCP — not via direct file writes — and offers a rule + guide follow-up. On a fresh machine, the first MCP call downloads the CLI (~5s) before proceeding.

If `/archcore:*` commands don't appear, check [Troubleshooting](/plugin/troubleshooting/).

## Uninstall

**Claude Code:**

```bash
/plugin uninstall archcore@archcore-plugins
```

To also clear the cached CLI binary:

```bash
rm -rf "$CLAUDE_PLUGIN_DATA/archcore"
# or, if CLAUDE_PLUGIN_DATA is unset:
rm -rf "${XDG_DATA_HOME:-$HOME/.local/share}/archcore-plugin"
```

**Cursor:** remove Archcore from plugin settings, and remove the `archcore` entry from your Cursor MCP config.
