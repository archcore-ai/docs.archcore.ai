The plugin targets AI coding agents that implement both Agent Skills and MCP as open standards. Skills, built-in agents, and MCP tools are shared across hosts. Only **hooks** and **plugin manifests** are host-specific.

## Support matrix

| Host             | Status       | Install                        | MCP wiring                                    | Hooks API                                               |
| ---------------- | ------------ | ------------------------------ | --------------------------------------------- | ------------------------------------------------------- |
| **Claude Code**  | Production   | Plugin marketplace             | Auto â€” plugin ships `.mcp.json` + launcher    | `SessionStart`, `PreToolUse`, `PostToolUse`             |
| **Cursor**       | Implemented  | Plugin marketplace / local dir | User-registered (points at plugin's launcher) | `sessionStart`, `preToolUse`, `postToolUse`, `afterMCPExecution` |

**Production** means the plugin is deployed and used day-to-day; bug fixes and new skills ship through the marketplace.

**Implemented** means every skill and agent works, and all documented hooks fire, but some host-level edge cases still surface occasionally. On Codex CLI, hooks are additionally gated by a Codex-side feature flag â€” see [Codex CLI](#codex-cli) below.

## Host-specific differences

The plugin keeps skills, agents, and MCP tools identical across hosts. The differences are at the edges:

### Plugin manifest

| Aspect           | Claude Code                            | Cursor                                   | Codex CLI                                |
| ---------------- | -------------------------------------- | ---------------------------------------- | ---------------------------------------- |
| Manifest file    | `.claude-plugin/plugin.json` (minimal) | `.cursor-plugin/plugin.json` (explicit)  | `.codex-plugin/plugin.json` (explicit)   |
| Field pointers   | Convention-based                       | Explicit `skills`, `agents`, `hooks`     | Explicit `commands`, `agents`, `hooks`, `mcpServers` |

### MCP server registration

| Aspect              | Claude Code                                       | Cursor                                                  | Codex CLI                                                |
| ------------------- | ------------------------------------------------- | ------------------------------------------------------- | -------------------------------------------------------- |
| Who registers MCP   | The plugin, via `.mcp.json` at plugin root        | The user, via Cursor MCP settings or project `mcp.json` | The plugin, via `.codex.mcp.json` referenced from manifest's `mcpServers` |
| Command points at   | `${CLAUDE_PLUGIN_ROOT}/bin/archcore mcp`          | Same launcher, or a globally-installed `archcore`       | `./bin/archcore mcp` with `cwd: "."` (Codex rebases relative paths to plugin install root) |
| CLI install step    | None â€” launcher auto-downloads on first use       | Either via the launcher path, or install CLI globally   | None â€” launcher auto-downloads on first use              |
| Duplicate handling  | Launcher defers to any global `archcore` on PATH  | N/A â€” user controls the registration                    | Launcher defers to any global `archcore` on PATH         |

### Hook files

| Aspect           | Claude Code                            | Cursor                                   | Codex CLI                                |
| ---------------- | -------------------------------------- | ---------------------------------------- | ---------------------------------------- |
| Hook file        | `hooks/hooks.json`                     | `hooks/cursor.hooks.json`                | `hooks/codex.hooks.json`                 |
| Trigger names    | `SessionStart`, `PreToolUse`, `PostToolUse` | `sessionStart`, `preToolUse`, `postToolUse`, `afterMCPExecution` | `SessionStart`, `PreToolUse`, `PostToolUse` |
| Write matcher    | `Write|Edit`                           | `Write` only                             | `Write|Edit|apply_patch`                 |
| Output protocol  | Exit codes + structured JSON           | JSON wrapper objects                     | Exit codes + structured JSON             |
| Path env var     | `${CLAUDE_PLUGIN_ROOT}`                | `${CURSOR_PLUGIN_ROOT}`                  | `${PLUGIN_ROOT}` (host-neutral)          |

All three hook files live in the same plugin package and are selected automatically by the host.

### Cursor: `afterMCPExecution`

Cursor exposes an extra `afterMCPExecution` trigger that fires after MCP tool calls succeed. The plugin uses it to run `validate-archcore`, the cascade staleness check, and the precision-warnings check in a single hook. On Claude Code and Codex CLI, the same work happens via `PostToolUse` with multiple matchers.

### Codex CLI

Codex CLI uses the same `PostToolUse` semantics as Claude Code, plus a third write surface â€” Codex's `apply_patch` tool â€” which is added to the `PreToolUse` matcher. Codex hooks are gated by a Codex-side feature flag:

```bash
codex features enable plugin_hooks
```

This flag is currently under development in Codex (default off). Without it, hooks are inert but skills, MCP, and agents still work normally.

The path env var is `${PLUGIN_ROOT}` â€” Codex's canonical, host-neutral name. There is no `${CODEX_PLUGIN_ROOT}` variable.

### Command surfaces

Skills are surfaced differently depending on host conventions:

- **Claude Code** and **Cursor** load skills directly from `skills/<name>/SKILL.md`.
- **Codex CLI** surfaces skills through thin `commands/<name>.md` wrappers (16 of them) â€” one per skill.

The skill content itself is the same; only the entry point differs.

### Agents

Both `archcore-assistant` and `archcore-auditor` ship in two file formats side by side under `agents/`:

- **`.md`** â€” consumed by Claude Code and Cursor.
- **`.toml`** â€” consumed by Codex CLI. The auditor TOML adds `sandbox_mode = "read-only"` plus a `disabled_tools` list covering the five mutating MCP tools; the assistant TOML uses `sandbox_mode = "workspace-write"`.

Tool whitelists are identical across formats â€” the TOML variants just translate the safety boundaries into Codex's native sandbox model.

## Choosing a host

**Use Claude Code if:**
- You want the most mature path â€” it has the deepest testing coverage.
- You're already invested in the Claude Code CLI workflow.

**Use Cursor if:**
- You prefer a full editor over a terminal-first agent.
- You want `afterMCPExecution` semantics (single post-MCP hook).

**Use Codex CLI if:**
- You want a terminal-first agent with auto-registered MCP (no manual wiring).
- You're comfortable enabling the `plugin_hooks` feature flag to get full hook coverage.

**All three are first-class.** Skills, tracks, agents, and guardrails behave identically.

## Unsupported hosts

The plugin is not available on GitHub Copilot yet, and on any other MCP-aware CLI without a plugin runtime. Those agents currently work with the [CLI path](/cli/overview/) only â€” you get MCP access and agent integrations, but no plugin skills or tracks.

Track planned host support in the plugin's [repository](https://github.com/archcore-ai/plugin).
