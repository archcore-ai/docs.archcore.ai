Archcore gives your AI coding agents git-native context — structured documents that persist across sessions and work with every agent.

## Install

<Tabs>
  <TabItem label="macOS / Linux">
    ```bash
    curl -fsSL https://archcore.ai/install.sh | bash
    ```
  </TabItem>
  <TabItem label="Windows">
    ```powershell
    irm https://archcore.ai/install.ps1 | iex
    ```

    Installs `archcore.exe` under `%LOCALAPPDATA%\Programs\archcore` and adds it to your user `PATH`. Open a new PowerShell window after install so the `PATH` change is picked up. Requires PowerShell 5.1+ (ships by default on Windows 10/11).
  </TabItem>
  <TabItem label="Windows (WSL)">
    If you prefer a Linux environment on Windows, install <a href="https://learn.microsoft.com/en-us/windows/wsl/install" target="_blank">WSL</a> and run inside it:

    ```bash
    curl -fsSL https://archcore.ai/install.sh | bash
    ```
  </TabItem>
</Tabs>

To update later, run `archcore update`.

## Initialize your project

```bash
cd your-project
archcore init
```

This does three things:

1. Creates `.archcore/` directory with `settings.json`
2. Auto-detects your coding agents (Claude Code, Cursor, Copilot, Gemini CLI, etc.)
3. Installs MCP server config and session hooks for each detected agent

**Verify:** Open a conversation with your agent and ask:

> "What Archcore documents exist in this project?"

The agent should respond that no documents exist yet. If it doesn't recognize the question, the MCP server may not be configured — check that `archcore init` detected your agent in its output. See the [troubleshooting guide](/reference/troubleshooting/) if needed.

## Create your first document

Ask your agent to create an ADR about a real decision in your project:

> "Create an ADR about using PostgreSQL as our primary database"

The agent calls the `create_document` MCP tool and generates a document with the right template, frontmatter, and sections.

**Verify:** Check what the agent created:

```bash
ls .archcore/
```

You should see a file like `use-postgres.adr.md`. Open it — the content follows this structure:

```markdown
---
title: Use PostgreSQL as Primary Database
status: draft
---

## Context

We need a reliable relational database for our application.

## Decision

Use PostgreSQL for all persistent data storage.

## Alternatives Considered

- MySQL — fewer advanced features (JSONB, arrays, CTEs)
- MongoDB — doesn't fit our relational data model

## Consequences

### Positive
- Strong ACID guarantees
- Excellent ecosystem and tooling

### Negative
- Requires schema migrations for changes
- Horizontal scaling is more complex than NoSQL
```

The key parts: YAML frontmatter with `title` and `status`, a `slug.type.md` filename, and template sections matching the ADR type.

:::tip
You can also create documents manually following the [document structure](/concepts/documents/), but letting the agent do it is faster and less error-prone.
:::

Run `archcore status` to confirm everything is structured correctly.

## Use with your agent

Start a new conversation and ask questions that require project context:

> "What architectural decisions have been made in this project?"

The agent calls `list_documents`, finds your ADR, and reports back. Go deeper:

> "Summarize our database decision and its trade-offs"

The agent reads the full document and gives you an informed answer drawn from your repo context — not from guesswork or generic knowledge.

**Verify:** The agent's response should reference specific details from your ADR: the alternatives you considered, the consequences you documented, the status. If it gives a generic answer about PostgreSQL instead, the MCP connection may not be active.

This is the core loop: you capture knowledge as structured documents, and every agent session has that context from the start.

## Build on it

Decisions produce rules. Rules need guides. Create a second document that connects to the first:

> "Create a rule for database migrations based on the PostgreSQL ADR"

The agent creates a `.rule.md` document with imperative statements about how your team handles schema changes. Then link them:

> "Link the migration rule to the PostgreSQL ADR"

The agent calls `add_relation` to connect the two documents. Any agent reading either document now sees the connection.

**Verify:** Ask *"Show me all documents and their relations."* You should see both documents and the relation between them — the beginning of a structured project context that grows with your codebase.

## Starting from an existing repo

Archcore is additive — it works alongside your existing files.

**You have a `CLAUDE.md` or `.cursorrules`:**
Keep it. Over time, extract structured knowledge into typed `.archcore/` documents. A coding standard becomes a `.rule.md`. A setup procedure becomes a `.guide.md`. Once knowledge lives in `.archcore/`, it works with every agent, not just one.

**You have `docs/` or existing ADRs:**
Ask your agent: *"Migrate our existing documentation into Archcore format."* It reads your files, picks the right document types, and creates structured documents in `.archcore/`.

**You have a wiki or Notion:**
Start by capturing the most frequently referenced decisions and rules as `.archcore/` documents. Migrate incrementally — start with what agents need most.
